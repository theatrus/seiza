# Operating the Seiza satellite mirror

The Seiza satellite mirror is a rolling set of epoch-appropriate TLE catalogs
for historical image analysis. It is intentionally separate from the complete
plate-solving bundle: solver catalogs are coherent versioned releases, while
orbital elements add one immutable time slice at a time.

The normal cadence is twice daily, at 00:00 and 12:00 UTC. One current
SatChecker response is about 4.08 MiB, so this publishes roughly 3.0 GB per
year before S3 replication. Runtime clients accept the nearest snapshot within
12 hours and fall back to the public IAU SatChecker endpoint when the mirror
does not cover an exposure.

## Published layout

The public base is `https://downloads.seiza.fyi/satellites/v1`:

```text
manifest.json
manifests/<generation>.json
artifacts/<sha256>/satellites.tle
```

`manifest.json` is the only mutable object. Every TLE object and archived
manifest is immutable. A manifest entry records the SatChecker query epoch,
upstream source URL, byte count, SHA-256, and content-addressed key. Readers
reject unknown schema fields, unsorted or duplicate epochs, non-content-
addressed keys, invalid timestamps, wrong sizes, and hash mismatches.

Schema version 1 is generated by the CLI rather than authored by hand:

```json
{
  "schema_version": 1,
  "generated_at_utc": "2026-07-19T12:20:00Z",
  "snapshots": [
    {
      "query_time_utc": "2025-10-18T12:00:00Z",
      "source_url": "https://satchecker.cps.iau.org/tools/tles-at-epoch/?epoch=2460967.00000000&format=txt",
      "key": "artifacts/<lowercase-sha256>/satellites.tle",
      "sha256": "<lowercase-sha256>",
      "size_bytes": 4274736
    }
  ]
}
```

Snapshots are strictly sorted by unique `query_time_utc`. Several epochs may
reference the same content hash. Artifacts are limited to 256 MiB and the
manifest to 16 MiB, comfortably above the current roughly 4.08 MiB response
size while still bounding a compromised or misconfigured publisher.

Publication order is mandatory:

1. Upload new immutable `artifacts/` objects.
2. Upload an immutable copy below `manifests/`.
3. Upload `manifest.json` last.

A failure before step 3 leaves every client on the previous complete
manifest. Never overwrite an object below `artifacts/`.

## Server directories

Use persistent directories; the cache is also useful for local reprocessing:

```text
/var/lib/seiza/satellite-mirror/
  cache/                  # seiza-satellites durable cache, default cap 5 GiB
  current-manifest.json   # last successfully published pointer
  next-manifest.json      # generated transaction candidate
  stage/                  # only new immutable objects for the next S3 sync
  publish.lock
```

The publishing account needs `GetObject` for the current manifest and
`PutObject` under the mirror prefix. It does not need permission outside
`satellites/v1/`.

## Retention policy

The public manifest is cumulative by default: a historical epoch remains
available after it has been published. Do not attach an S3 expiration rule to
`artifacts/` unless the same entries are deliberately removed from a future
manifest first. Content addressing deduplicates identical upstream payloads.

The publisher's local cache is different. It uses the shared Seiza satellite
cache ceiling (5 GiB by default), so old local snapshots are pruned while the
downloaded base manifest retains the complete public inventory. At the current
twice-daily cadence, 5 GiB holds roughly 1,250 typical snapshots, or about 20
months. The next manifest only stages hashes that the base manifest does not
already reference.

## One publication

The cron job performs two explicit phases. First, ask the same provider
resolver used by applications for an exact origin snapshot. `--origin` is the
publisher policy: it bypasses the mirror being updated and the normal 12-hour
nearby-cache window, while still reusing an exact validated cache hit:

```bash
seiza download-data satellite-history \
  --epoch 2025-10-18T12:00:00Z \
  --cache /var/lib/seiza/satellite-mirror/cache \
  --origin
```

Then roll every validated cache entry into the schema and stage only artifacts
that are not already referenced by the previous manifest:

```bash
seiza build-data satellite-manifest \
  --cache /var/lib/seiza/satellite-mirror/cache \
  --base-manifest /var/lib/seiza/satellite-mirror/current-manifest.json \
  --output /var/lib/seiza/satellite-mirror/next-manifest.json \
  --artifact-dir /var/lib/seiza/satellite-mirror/stage
```

Omit `--base-manifest` for the first publication. The generator parses every
new TLE, recomputes its SHA-256 and size, validates the rolled manifest, writes
the new pointer atomically, and creates the same content-addressed S3 layout
that the runtime resolver expects.

## Ready-to-install publisher

Install this as `/usr/local/sbin/publish-seiza-satellites`. Pass the exact UTC
bucket as its only argument.

```bash
#!/usr/bin/env bash
set -euo pipefail

EPOCH=${1:?pass an RFC3339 UTC epoch such as 2025-10-18T12:00:00Z}
ROOT=${SEIZA_SATELLITE_ROOT:-/var/lib/seiza/satellite-mirror}
S3_BASE=${SEIZA_SATELLITE_S3_BASE:?set an s3://bucket/prefix base}
CACHE="$ROOT/cache"
CURRENT="$ROOT/current-manifest.json"
NEXT="$ROOT/next-manifest.json"
STAGE="$ROOT/stage"

mkdir -p "$CACHE" "$ROOT"
exec 9>"$ROOT/publish.lock"
flock -n 9 || exit 0

rm -rf "$STAGE"
mkdir -p "$STAGE/artifacts"

if [[ ! -f "$CURRENT" ]]; then
  REMOTE_BASE="$ROOT/remote-base.json"
  rm -f "$REMOTE_BASE"
  if aws s3 cp "$S3_BASE/satellites/v1/manifest.json" "$REMOTE_BASE" --no-progress; then
    mv "$REMOTE_BASE" "$CURRENT"
  else
    rm -f "$REMOTE_BASE"
    if [[ ${SEIZA_SATELLITE_INITIAL_PUBLISH:-0} != 1 ]]; then
      echo "Cannot load the published base manifest; refusing to replace its history." >&2
      echo "Set SEIZA_SATELLITE_INITIAL_PUBLISH=1 only for the first publication." >&2
      exit 1
    fi
  fi
fi

seiza download-data satellite-history --epoch "$EPOCH" --cache "$CACHE" --origin

BUILD_ARGS=(
  build-data satellite-manifest
  --cache "$CACHE"
  --output "$NEXT"
  --artifact-dir "$STAGE"
)
if [[ -f "$CURRENT" ]]; then
  BUILD_ARGS+=(--base-manifest "$CURRENT")
fi
seiza "${BUILD_ARGS[@]}"

aws s3 sync "$STAGE/artifacts/" \
  "$S3_BASE/satellites/v1/artifacts/" \
  --cache-control 'public,max-age=31536000,immutable' \
  --no-progress

GENERATION=$(jq -r '.generated_at_utc' "$NEXT" | tr ':.' '--')
aws s3 cp "$NEXT" \
  "$S3_BASE/satellites/v1/manifests/$GENERATION.json" \
  --content-type application/json \
  --cache-control 'public,max-age=31536000,immutable' \
  --no-progress

aws s3 cp "$NEXT" \
  "$S3_BASE/satellites/v1/manifest.json" \
  --content-type application/json \
  --cache-control 'public,max-age=300,must-revalidate' \
  --no-progress

mv "$NEXT" "$CURRENT"
```

For a twice-daily cron, percent signs must be escaped in crontab:

```cron
20 0,12 * * * SEIZA_SATELLITE_S3_BASE=s3://YOUR-BUCKET/YOUR-PREFIX /usr/local/sbin/publish-seiza-satellites "$(date -u +\%Y-\%m-\%dT\%H:00:00Z)" >>/var/log/seiza-satellite-mirror.log 2>&1
```

Running at 20 minutes past the bucket gives the upstream service time to
publish nearby elements. The job is idempotent: cache hits do not redownload,
content hashes deduplicate identical TLE payloads, S3 sync skips existing
objects, and the manifest can safely be regenerated and republished.

For the very first publication, set `SEIZA_SATELLITE_INITIAL_PUBLISH=1` for
that run. Later runs deliberately fail if neither the local nor S3 base
manifest can be read; silently replacing a cumulative manifest with only the
bounded local cache would discard older public history.

## Backfill

Supply several observing epochs to prewarm older nights before generating the
next manifest:

```bash
seiza download-data satellite-history \
  --epoch 2025-10-17T12:00:00Z 2025-10-18T12:00:00Z \
  --cache /var/lib/seiza/satellite-mirror/cache \
  --origin
```

Keep backfills bounded. The command is not intended to crawl SatChecker's
entire backend. After prewarming, run the same manifest build and S3 publication
transaction.

## Verification

After publishing, validate the public pointer and newest object:

```bash
BASE=https://downloads.seiza.fyi/satellites/v1
curl -fsS "$BASE/manifest.json" -o /tmp/seiza-satellite-manifest.json
KEY=$(jq -r '.snapshots[-1].key' /tmp/seiza-satellite-manifest.json)
EXPECTED=$(jq -r '.snapshots[-1].sha256' /tmp/seiza-satellite-manifest.json)
curl -fsS "$BASE/$KEY" -o /tmp/seiza-satellites.tle
printf '%s  %s\n' "$EXPECTED" /tmp/seiza-satellites.tle | sha256sum -c -
```

Finally run an application lookup for an exposure near the new epoch and
confirm its provenance reports `seiza_mirror`. If the public manifest or
artifact is missing or invalid, the resolver uses IAU SatChecker instead; it
never treats an incomplete mirror publication as clean satellite evidence.
